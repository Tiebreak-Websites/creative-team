# Deploy — cloud host with push-to-`prod` auto-deploy

The live site runs as **one Docker container** (the FastAPI backend serves both the
API and the built React SPA — single origin) on a cloud host that **redeploys
automatically on every push to the `prod` branch**. No machine of yours is involved,
nothing to install locally, and the app's own email/password login is the security.

```
git push → prod ──► Render builds the Dockerfile ──► live at https://<name>.onrender.com
                                                      backend serves /api/* + the SPA
                                                      app login gates everything (HTTPS, Secure cookie)
```

> **Why not Cloudflare for the app?** Cloudflare Workers/Pages are serverless JS/WASM
> — they can't run this Python backend (long-running OpenAI jobs, Pillow, subprocess
> Figma scripts, on-disk artifacts). They can host only the static frontend. Running
> the *whole* app in one container is simpler and keeps the functionality working.
> (If you connected a Cloudflare "Workers Builds" Git integration, disconnect it —
> see the last section — or its build will keep failing on every push.)

The repo ships the deploy config: [`Dockerfile`](../Dockerfile), [`render.yaml`](../render.yaml),
[`.dockerignore`](../.dockerignore).

## One-time setup (Render)

### 1. Pick the admin password

Generate a strong password + its bcrypt hash (you log in with the password; the
**hash** goes into Render — never the plaintext):

```powershell
cd platform\backend
.\.venv\Scripts\python.exe -c "import secrets; from passlib.context import CryptContext; pw=secrets.token_urlsafe(12); print('PASSWORD', pw); print('HASH', CryptContext(schemes=['bcrypt']).hash(pw))"
```

### 2. Create the service from the Blueprint

1. Push the `prod` branch to GitHub (already done if you followed the chat).
2. [Render](https://dashboard.render.com) → **New → Blueprint** → connect the
   `chr1srusevv/creative-team` repo. Render reads [`render.yaml`](../render.yaml) and
   proposes a free web service named **creative-team** that builds the Dockerfile and
   tracks the **`prod`** branch.
3. Click **Apply**.

### 3. Set the secret env vars

In the service's **Environment** tab, fill the `sync:false` keys (the Blueprint
declared them but left them blank):

| Key | Value |
| --- | --- |
| `ADMIN_PASSWORD_HASH` | the **HASH** from step 1 |
| `OPENAI_API_KEY` | your OpenAI key (Banner Builder) |
| `ANTHROPIC_API_KEY` | your Anthropic key (AI-assist / Creative Summary) — optional |
| `FIGMA_API_KEY` | your Figma token (Figma QA / Translate) — optional |

`PLATFORM_SECRET_KEY` is auto-generated by Render; `ADMIN_EMAIL`,
`PLATFORM_COOKIE_SECURE=true`, and `PLATFORM_DOCS=false` come from the Blueprint.

### 4. Deploy + log in

Render builds and deploys. When it's live, open
`https://creative-team.onrender.com` → the app's **login** page → sign in with
`ADMIN_EMAIL` + your password.

> Free tier note: the service spins down after ~15 min idle, so the first request
> after a nap takes ~30–60s to wake. Fine for internal use; upgrade the plan to keep
> it always-on.

## The `prod` workflow

`prod` is the live branch. To release:

```powershell
git checkout prod
git merge main          # bring in reviewed work
git push                # Render auto-deploys
```

Day-to-day: do work on a feature branch → PR into `main` → when you want it live,
merge `main` into `prod` and push. Every push to `prod` triggers a deploy; watch it
in the Render dashboard.

## Security (what protects the live site)

- **App login** — every route except the login + the Figma plugin bridge requires a
  valid session cookie (JWT, ~8h). Set a strong `ADMIN_PASSWORD_HASH` (step 1).
- **HTTPS + Secure cookie** — Render terminates TLS; `PLATFORM_COOKIE_SECURE=true`
  makes the session cookie `Secure`.
- **No API docs** — `PLATFORM_DOCS=false` hides `/docs` + `/openapi.json`; an
  anonymous visitor only ever sees the login page.
- **Stable signing key** — `PLATFORM_SECRET_KEY` (Render-generated) so sessions
  survive restarts and aren't a hardcoded value.

Want a network gate on top later (only your team can even load the page)? Put a
custom domain on Cloudflare and add a Cloudflare Access policy in front — but that
needs a domain, so it's a later step.

## Other hosts (same Dockerfile)

The image is host-agnostic — it just needs `$PORT`:

- **Railway** — New Project → Deploy from GitHub repo → it detects the Dockerfile;
  set the branch to `prod` and add the same env vars.
- **Fly.io** — `fly launch --dockerfile Dockerfile` (it generates `fly.toml`), set
  secrets with `fly secrets set ...`, and wire a small GitHub Action to run
  `fly deploy` on push to `prod`.

## Alternative: self-host behind a Cloudflare Tunnel

If you ever want to run it from a machine you control and expose it via Cloudflare
(custom domain, Cloudflare Access), the scaffold is in [`deploy/`](deploy)
(`cloudflared/config.example.yml`, `.env.production.example`) and the steps are in the
git history of this file. Note: this needs a network that allows `cloudflared`'s
egress (UDP/TCP port 7844) — a locked-down corporate network will block it.
